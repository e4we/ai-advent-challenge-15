**1. Валидация плана (перед выполнением)** Проверяем структуру плана после парсинга JSON: есть ли summary, не пустые ли шаги, указаны ли tool names, существуют ли упомянутые инструменты в реестре, нет ли дублей. Чисто программная проверка, без вызова LLM.

**2. Валидация результатов tool call (во время выполнения)** После каждого вызова инструмента проверяем результат: команда вернула ненулевой exit code, файл не записался, grep ничего не нашёл. Если ошибка — передаём LLM не просто результат, а результат + пометку "это ошибка, реши что делать". Тоже программная, без лишних вызовов LLM.

**3. Валидация шага через LLM (после каждого шага)** Отдельный запрос к LLM: "Вот шаг плана: '...'. Вот что было сделано: '...'. Шаг выполнен успешно? Ответь JSON: {success: true/false, reason: '...'}". Дорого — каждый шаг = дополнительный API-вызов, но ловит смысловые ошибки.

**4. Пост-валидация плана (после выполнения всех шагов)** Один запрос к LLM в конце: "Вот план, вот что было сделано. Всё ли выполнено корректно?". Дешевле чем вариант 3 (один вызов вместо N), но не ловит проблемы по ходу.

**5. Автотесты как валидация** Если в плане есть шаг "написать код" — агент автоматически добавляет шаг "запустить и проверить" (go build, go test, python -c "import module"). Не требует лишних LLM-вызовов, ловит реальные ошибки.

Варианты 1 и 2 — бесплатные (без лишних API-запросов) и закрывают 80% проблем. Варианты 3–5 добавляют надёжность, но стоят денег/времени. 

---

Вариант 1. Пример для плана, который вернул LLM:

json

````json
{
  "summary": "",
  "steps": [
    {"number": 1, "action": "Прочитать файл", "tool": "read_file"},
    {"number": 1, "action": "", "tool": "magic_tool"},
    {"number": 3, "action": "Удалить всё", "tool": ""}
  ],
  "risks": []
}
```

Валидатор проверяет и возвращает список проблем:
```
❌ Ошибки плана:
  • summary пустой — LLM не описал задачу
  • шаг 1: дублирующийся номер (два шага с number=1)
  • шаг 2: пустой action — непонятно что делать
  • шаг 2: инструмент "magic_tool" не существует
      (доступны: run_command, read_file, write_file, list_files)
  • шаг 3: не указан инструмент
````

Если ошибки есть — план не показывается пользователю, а отправляется обратно LLM с просьбой исправить. Если после повторной попытки план всё ещё невалидный — показываем пользователю как есть с пометкой о проблемах.

Вариант 2. Пример: агент выполняет план из трёх шагов. После каждого tool call валидатор проверяет результат:

```
🤖 Шаг 1: читаю конфиг проекта
  🔨 Инструмент: read_file(path="config.yaml")
  📋 Результат: ❌ Ошибка чтения: open config.yaml: no such file or directory
  ⚠️  Валидация: файл не найден → сообщаю LLM как ошибку

     LLM получает: {tool_result: "...", is_error: true}
     и сам решает: попробовать другой путь, спросить, или пропустить шаг

🤖 Понял, попробую config.yml...
  🔨 Инструмент: read_file(path="config.yml")
  📋 Результат: port: 8080\ndb: postgres://...
  ✅ Валидация: файл прочитан (непустой результат)

🤖 Шаг 2: создаю main.go
  🔨 Инструмент: write_file(path="main.go", content="...")
  📋 Результат: ✅ Файл записан: main.go (42 строки)
  ✅ Валидация: файл записан успешно

🤖 Шаг 3: проверяю компиляцию
  🔨 Инструмент: run_command(command="go build .")
  📋 Результат: ./main.go:15:2: undefined: handleRequest
  ⚠️  Валидация: команда вернула exit code 1 → сообщаю LLM как ошибку

     LLM получает: {tool_result: "...", is_error: true}
     и сам решает: исправить код и попробовать снова
```

Суть: валидатор не решает что делать с ошибкой — он только определяет "это ошибка или нет" по объективным признакам (exit code, пустой файл, "no such file") и ставит флаг `is_error: true`. Дальше LLM сам разбирается. Без дополнительных API-вызовов.

Вариант 3. Пример: агент выполняет план из трёх шагов. После каждого шага отдельный запрос к LLM проверяет, действительно ли шаг выполнен корректно.

```
🤖 Шаг 1: прочитать конфиг проекта
  🔨 Инструмент: read_file(path="config.yaml")
  📋 Результат: port: 8080\ndb: postgres://localhost/myapp

  🔍 Валидация шага 1...
     Запрос к LLM: "Шаг плана: 'прочитать конфиг проекта'.
                    Результат: port: 8080, db: postgres://...
                    Шаг выполнен успешно? Ответь JSON."
     Ответ: {"success": true, "reason": "Конфиг прочитан, содержит порт и строку подключения к БД"}
  ✅ Шаг 1 валидирован

  📊 Прогресс: [██████░░░░░░░░░░░░░░] 1/3

🤖 Шаг 2: создать main.go с HTTP-сервером
  🔨 Инструмент: write_file(path="main.go", content="package main...")
  📋 Результат: ✅ Файл записан: main.go (35 строк)

  🔍 Валидация шага 2...
     Запрос к LLM: "Шаг плана: 'создать main.go с HTTP-сервером'.
                    Результат: файл записан, 35 строк.
                    Содержимое: package main... http.ListenAndServe...
                    Шаг выполнен успешно?"
     Ответ: {"success": true, "reason": "Файл создан, содержит HTTP-сервер на порту из конфига"}
  ✅ Шаг 2 валидирован

  📊 Прогресс: [████████████░░░░░░░░] 2/3

🤖 Шаг 3: проверить компиляцию
  🔨 Инструмент: run_command(command="go build .")
  📋 Результат: ./main.go:12:5: undefined: loadConfig

  🔍 Валидация шага 3...
     Запрос к LLM: "Шаг плана: 'проверить компиляцию'.
                    Результат: ./main.go:12:5: undefined: loadConfig
                    Шаг выполнен успешно?"
     Ответ: {"success": false, "reason": "Компиляция провалилась — функция loadConfig не определена"}
  ❌ Шаг 3 не пройден: Компиляция провалилась — функция loadConfig не определена

     → Отправляю LLM: "Шаг 3 не прошёл валидацию: компиляция провалилась —
        функция loadConfig не определена. Исправь и попробуй снова."

🤖 Понял, добавляю функцию loadConfig...
  🔨 Инструмент: edit_file(path="main.go", old_text="...", new_text="func loadConfig()...")
  🔨 Инструмент: run_command(command="go build .")
  📋 Результат: (пусто — компиляция успешна)

  🔍 Повторная валидация шага 3...
     Ответ: {"success": true, "reason": "Компиляция прошла успешно после исправления"}
  ✅ Шаг 3 валидирован

  📊 Прогресс: [████████████████████] 3/3
  ✅ Все шаги плана выполнены!
```

Суть: после каждого шага — дополнительный API-вызов к LLM с вопросом "шаг выполнен?". LLM-валидатор получает описание шага из плана и фактический результат, сравнивает их по смыслу. Если `success: false` — основному LLM отправляется причина провала с просьбой исправить. Дорого (N шагов = N дополнительных запросов), но ловит смысловые ошибки, которые программная валидация пропустит.

Вариант 4. Пример: агент выполнил план из четырёх шагов. После завершения всех шагов — один запрос к LLM для проверки всего сразу.

```
🤖 Шаг 1: посмотреть структуру проекта
  🔨 Инструмент: list_files(directory=".", recursive=true)
  📋 Результат: main.go, go.mod, handlers/

🤖 Шаг 2: прочитать main.go
  🔨 Инструмент: read_file(path="main.go")
  📋 Результат: package main... (48 строк)

🤖 Шаг 3: добавить middleware для логирования
  🔨 Инструмент: edit_file(path="main.go", old_text="...", new_text="...")
  📋 Результат: ✅ заменено 3→12 строк

🤖 Шаг 4: проверить компиляцию
  🔨 Инструмент: run_command(command="go build .")
  📋 Результат: (пусто — успешно)

  📊 Прогресс: [████████████████████] 4/4

🤖 Готово! Добавил middleware для логирования всех запросов.

  ─────────────────────────────────────────
  🔍 Пост-валидация... (проверяю весь план целиком)

     Запрос к LLM:
       "Вот план, который нужно было выполнить:
        1. Посмотреть структуру проекта
        2. Прочитать main.go
        3. Добавить middleware для логирования
        4. Проверить компиляцию

        Вот что было сделано:
        - list_files → main.go, go.mod, handlers/
        - read_file main.go → 48 строк, HTTP-сервер
        - edit_file main.go → добавлено 9 строк (func loggingMiddleware)
        - go build → успешно

        Проверь:
        1. Все ли шаги выполнены?
        2. Соответствуют ли результаты целям шагов?
        3. Есть ли пропущенные или некорректные действия?
        Ответь JSON."

     Ответ:
     {
       "all_steps_completed": true,
       "issues": [
         {
           "step": 3,
           "severity": "warning",
           "message": "Middleware добавлен, но не подключён к роутеру — нужен вызов router.Use(loggingMiddleware)"
         }
       ],
       "summary": "3 из 4 шагов выполнены корректно. Шаг 3 выполнен частично: код middleware написан, но не подключён."
     }

  ⚠️  Пост-валидация нашла проблему:
     Шаг 3: Middleware добавлен, но не подключён к роутеру

  → Отправляю LLM: "Пост-валидация выявила проблему: middleware
     написан, но не подключён. Добавь router.Use(loggingMiddleware)."

🤖 Верно, исправляю.
  🔨 Инструмент: edit_file(path="main.go", ...)
  📋 Результат: ✅ добавлен вызов router.Use(loggingMiddleware)

  🔍 Повторная пост-валидация...
     Ответ: {"all_steps_completed": true, "issues": [], "summary": "Все 4 шага выполнены корректно."}

  ✅ Пост-валидация пройдена

╔══ 📊 ИТОГ ═══════════════════════════════════╗
║  ✅ 1. Посмотреть структуру проекта
║  ✅ 2. Прочитать main.go
║  ✅ 3. Добавить middleware для логирования
║  ✅ 4. Проверить компиляцию
╠══════════════════════════════════════════════╣
║  ✅ Все 4 шагов выполнены! (валидация пройдена)
╚══════════════════════════════════════════════╝
```

Разница с вариантом 3: один API-вызов вместо четырёх. Валидатор видит всю картину целиком, поэтому может поймать проблемы связанности между шагами (как в примере — middleware написан, но не подключён). Вариант 3 проверял бы шаг 3 изолированно и мог бы пропустить это, потому что файл формально записан. Минус — если ошибка была на шаге 1, агент узнает об этом только после выполнения всех четырёх шагов.

Вариант 5. Пример: агент выполняет план из трёх шагов. В плане есть шаг "написать код" — агент автоматически проверяет результат реальным запуском.

```
🤖 Шаг 1: создать Go-модуль и main.go с HTTP-сервером
  🔨 Инструмент: run_command(command="go mod init myserver")
  📋 Результат: go: creating new go.mod
  🔨 Инструмент: write_file(path="main.go", content="package main...")
  📋 Результат: ✅ Файл записан: main.go (52 строки)

  🧪 Автотест: обнаружен .go файл → проверяю компиляцию
     run_command: go build .
     ✅ Компиляция успешна

  📊 Прогресс: [██████░░░░░░░░░░░░░░] 1/3

🤖 Шаг 2: добавить обработчики и тесты
  🔨 Инструмент: write_file(path="handlers.go", content="package main...")
  📋 Результат: ✅ Файл записан: handlers.go (38 строк)
  🔨 Инструмент: write_file(path="handlers_test.go", content="package main...")
  📋 Результат: ✅ Файл записан: handlers_test.go (27 строк)

  🧪 Автотест: обнаружен _test.go → запускаю тесты
     run_command: go test ./...
     ❌ --- FAIL: TestCreateTodo (0.00s)
        handlers_test.go:15: expected status 201, got 200

  ⚠️  Тесты не прошли → отправляю LLM:
     "Автотест провалился: TestCreateTodo ожидает статус 201, но
      обработчик возвращает 200. Исправь обработчик или тест."

🤖 Исправляю — обработчик должен возвращать 201 Created.
  🔨 Инструмент: edit_file(path="handlers.go", old_text="w.WriteHeader(200)", new_text="w.WriteHeader(201)")
  📋 Результат: ✅ заменено

  🧪 Повторный автотест:
     run_command: go test ./...
     ✅ ok  myserver 0.003s

  📊 Прогресс: [████████████░░░░░░░░] 2/3

🤖 Шаг 3: создать Dockerfile
  🔨 Инструмент: write_file(path="Dockerfile", content="FROM golang:1.22...")
  📋 Результат: ✅ Файл записан: Dockerfile (12 строк)

  🧪 Автотест: обнаружен Dockerfile → проверяю синтаксис
     run_command: docker build --check .
     ✅ Синтаксис валиден

  📊 Прогресс: [████████████████████] 3/3
  ✅ Все шаги плана выполнены!

╔══ 📊 ИТОГ ═══════════════════════════════════╗
║  ✅ 1. Создать Go-модуль и main.go     🧪 build ok
║  ✅ 2. Добавить обработчики и тесты    🧪 tests ok (2-я попытка)
║  ✅ 3. Создать Dockerfile              🧪 syntax ok
╠══════════════════════════════════════════════╣
║  ✅ Все 3 шагов выполнены! (3 автотеста пройдено)
╚══════════════════════════════════════════════╝
```

Суть: после каждого tool call `OnToolResult` смотрит что было создано/изменено и подбирает проверку по типу файла:

|Файл|Автотест|
|---|---|
|`*.go`|`go build .`|
|`*_test.go`|`go test ./...`|
|`*.py`|`python -c "import module"` или `python -m pytest`|
|`*.js/ts`|`node -c file.js` или `npm test`|
|`Dockerfile`|`docker build --check .`|
|`*.yaml`|`python -c "import yaml; yaml.safe_load(open('file'))"`|
|`Makefile`|`make -n` (dry run)|

Ноль API-вызовов. Ловит реальные ошибки (не компилируется, тесты падают), а не галлюцинации LLM о том что "всё работает". Минус — покрывает только файловые артефакты, не проверяет логическую корректность (для этого нужны варианты 3 или 4).
