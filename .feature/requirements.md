Для режима планирования:
Валидация 1 — проверка структуры плана (программная, перед показом пользователю)
Валидация 2 — проверка результата tool call (программная, после каждого инструмента)
Валидация 5 — автотесты по типу файла (программная, после записи/изменения файлов)

Схема:

"planning"
    │
    │ CreatePlan()
    ▼
┌──────────────────┐
│ ВАЛИДАЦИЯ 1      │  Программная проверка структуры:
│ ValidatePlan     │  summary? шаги? tools существуют? дубли?
└──────┬───────────┘
       │ ok                    │ fail
       ▼                      ▼
  "reviewing"          перегенерировать план
       │                (LLM получает список ошибок)
       │ Approve()
       ▼
  "executing"
       │
       │ tool call завершён
       ▼
┌──────────────────┐
│ ВАЛИДАЦИЯ 2      │  Программная проверка результата:
│ ValidateResult   │  exit code != 0? "no such file"?
└──────┬───────────┘  "permission denied"? пустой результат?
       │ ok                    │ fail
       ▼                      ▼
  продолжаем          LLM получает result + is_error: true
       │                (сам решает: повторить, пропустить, спросить)
       │
       │ write_file / edit_file / run_command завершён
       ▼
┌──────────────────┐
│ ВАЛИДАЦИЯ 5      │  Автотест по типу артефакта:
│ AutoTest         │  .go → go build
└──────┬───────────┘  _test.go → go test
       │ ok           .py → python -c "import ..."
       ▼              Dockerfile → docker build --check
  продолжаем
       │              │ fail
       │              ▼
       │         LLM получает вывод ошибки
       │           (сам решает: исправить код, другой подход)
       │
       │ все шаги done (трекер)
       ▼
    "done"

Три шлагбаума, ноль API-вызовов. Валидация 1 срабатывает один раз до executor'а, валидации 2 и 5 — внутри executor'а на каждой итерации через хуки OnToolResult.
